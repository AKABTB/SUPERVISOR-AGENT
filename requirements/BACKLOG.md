## 明细

### 1 · Bot 目标解析优化 —— 自然语言设目标

> **用户故事**：作为一名用户，我希望可以通过自然语言的形式进行目标设定，减少标准化格式的输入消耗。

| 项 | 内容 |
|---|---|
| **现状** | 目标要走命令 + 固定分隔格式（如 `/goal 标题 \| 每天20:00`），靠竖线/空格/"每"字切分。 |
| **想要** | 直接说人话，例如"我要每天晚上八点前交健身打卡"，Bot 自动解析出「标题 + 频率 + 截止」。 |
| **验收要点** | ① 一句自然语言能拆出 标题 / cadence / deadline 三要素；② 拆不准时反问确认，不瞎猜落库；③ 解析完即落库、回到督促本职，不滑向"陪你聊怎么定目标"的发散。 |
| **大概动哪里** | `supervisor/core/scheduler.py`（cadence 解析）、`supervisor/core/intent.py`（LLM 判断范式）；`supervisor/interfaces/telegram_bot.py` 的 `_do_goal` 入口接自然语言分支。 |

---

### 2 · Bot 目标执行优化 —— 执行过程有记忆

> **用户故事**：作为一名用户，我希望在目标执行时具备记忆，方便执行进度的精确判断。

| 项 | 内容 |
|---|---|
| **现状** | 交付审查基本是"单次对照目标挑遗漏"，跨多次提交缺少连续进度记忆。交付记录存在 `delivery_logs` 表，但审查时未必充分利用历史。 |
| **想要** | 审查/督促时能记住"这个目标之前交过什么、还差什么"，判断进度更精确。 |
| **验收要点** | ① 同一目标多次交付累积成"进度视图"，审查参考历史而非孤立判断；② 能识别"上次说要补的这次补了没"；③ 存储层可能要扩展进度/状态字段。 |
| **大概动哪里** | `supervisor/core/reviewer.py`（审查带入历史）；`supervisor/storage/`（`models.py` / `db.py`，加进度字段或聚合查询）；数据表 `delivery_logs`。 |

### 3 · Bot 目标拆分 —— 大目标拆子目标

> **用户故事**：作为一名用户，我希望可以创建一个大目标并支持进行拆分，方便复杂目标的管理。

| 项 | 内容 |
|---|---|
| **现状** | 目标是扁平单层结构（`goals` 表无父子关系），只有"主攻 / 排队"的优先级之分。 |
| **想要** | 一个大目标下能挂若干子目标/里程碑，复杂事情可分解管理；督促和进度可落到子目标粒度。 |
| **验收要点** | ① 目标支持父子层级（或里程碑列表）；② "只催主攻"规则在层级下怎么运作要想清楚（催大目标还是催当前活跃子目标）；③ Web 控制台能展示层级、增删改子目标。 |
| **大概动哪里** | `supervisor/storage/models.py`（Goal 加 parent_id 或子目标关系）+ `db.py`；`supervisor/core/project_manager.py`（主攻/督促逻辑适配层级）；`web/frontend/src/App.vue`（层级展示）。**结构性升级，动手前先对齐设计。** |

### 4 · 多端通知渠道优化 —— 可自选通知渠道

> **用户故事**：作为一名用户，我希望可以自己设定不同的通知渠道，方便在不同场景下接到任务。

| 项 | 内容 |
|---|---|
| **现状** | 通知只走 Telegram Bot 主动推送（`telegram_bot.py` 的 JobQueue）。部分网络环境下 Telegram 后台推送不稳，单渠道有风险。 |
| **想要** | 可配置多种通知渠道（邮件、企业微信、短信、系统推送等），按场景选择用哪个/哪几个。 |
| **验收要点** | ① 渠道可在配置或 Web 里增删、启用/停用；② 一次督促可发到一个或多个已启用渠道；③ 契合分层：通知作为"输出适配"的扩展，核心层只管"该催了"，不关心走哪个渠道。 |
| **大概动哪里** | 新增通知适配层（类比 `supervisor/llm/` 的可切换思路：抽象 Notifier 接口 + 各渠道实现 + 工厂）；`config.yaml` / Web 加渠道配置；`telegram_bot.py` 催收回调改为调统一通知层。 |
